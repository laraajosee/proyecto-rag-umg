1. DESCRIPCION GENERAL DEL PROYECTO
================================================================================

Este proyecto implementa un sistema de Retrieval-Augmented Generation (RAG) 
desarrollado como proyecto academico en la Universidad Mariano Galvez de 
Guatemala (UMG), ciclo 2026.

RAG es una arquitectura de inteligencia artificial que combina dos procesos:
  - RECUPERACION: Busqueda semantica de fragmentos relevantes de documentos 
    almacenados en una base de datos vectorial.
  - GENERACION: Un modelo de lenguaje (LLM) genera respuestas fundamentadas
    en los fragmentos recuperados, citando las fuentes originales.

Objetivo principal: Construir un sistema que responda preguntas basandose
exclusivamente en documentos PDF propios, eliminando las "alucinaciones" 
tipicas de los modelos de lenguaje.

4. ESQUEMA DE BASE DE DATOS (init.sql)
================================================================================

Archivo: init.sql

Este script SQL se ejecuta automaticamente al inicializar el contenedor
PostgreSQL por primera vez.

EXTENSION:
  CREATE EXTENSION IF NOT EXISTS vector;
  -- Habilita el tipo de dato 'vector' y las funciones de busqueda vectorial

TABLA PRINCIPAL: vector_store
  Campos:
    - id         UUID          PRIMARY KEY  DEFAULT gen_random_uuid()
    - content    TEXT                        -- Contenido del chunk de texto
    - metadata   JSONB                      -- Metadatos: fuente, chunk_index, etc.
    - embedding  VECTOR(768)                -- Vector de embedding (768 dimensiones)

INDICE DE BUSQUEDA VECTORIAL:
  CREATE INDEX IF NOT EXISTS vector_store_embedding_idx 
  ON vector_store 
  USING ivfflat (embedding vector_cosine_ops) 
  WITH (lists = 100);

  Detalles del indice:
    - Tipo: IVFFlat (Inverted File Flat)
    - Operador: vector_cosine_ops (similitud coseno)
    - Listas: 100 (particiones para busqueda aproximada)
    - Proposito: Acelerar busquedas de similitud semantica

TABLA DE METRICAS: evaluation_metrics
  Campos:
    - id              SERIAL    PRIMARY KEY
    - experiment_name VARCHAR(100)     -- Nombre del experimento
    - query_text      TEXT             -- Consulta evaluada
    - precision_score DOUBLE PRECISION -- Precision del resultado
    - recall_score    DOUBLE PRECISION -- Recall del resultado
    - latency_ms      BIGINT           -- Latencia en milisegundos
    - top_k           INTEGER          -- Numero de chunks recuperados
    - chunk_size      INTEGER          -- Tamano del chunk utilizado
    - created_at      TIMESTAMP        DEFAULT CURRENT_TIMESTAMP

CONSULTA DE METRICAS PROMEDIO:
  SELECT experiment_name, 
         AVG(precision_score) AS avg_precision,
         AVG(recall_score) AS avg_recall, 
         AVG(latency_ms) AS avg_latency
  FROM evaluation_metrics 
  GROUP BY experiment_name;

6. ARQUITECTURA DEL SISTEMA
================================================================================

El sistema sigue la arquitectura RAG (Retrieval-Augmented Generation) con
el siguiente flujo:

FLUJO DE INGESTA (Carga de documentos):
  [PDFs] -> [DocumentIngestionService] -> [Chunking] -> [OpenAI Embeddings] -> [pgvector]

  1. Los archivos PDF se colocan en: src/main/resources/documents/
  2. El servicio de ingesta lee cada PDF
  3. Los documentos se dividen en fragmentos (chunks) de tamano configurable
  4. Cada chunk se convierte en un vector de 768 dimensiones via OpenAI
  5. Los vectores se almacenan en la tabla vector_store de PostgreSQL

FLUJO DE CONSULTA (Preguntas del usuario):
  [Query] -> [QueryController] -> [RAGService] -> [pgvector Search] -> [OpenAI GPT] -> [Response]

  1. El usuario envia una pregunta via POST /api/query
  2. La pregunta se convierte en un vector de embedding
  3. Se buscan los top-K chunks mas similares en pgvector (similitud coseno)
  4. Los chunks recuperados se envian como contexto al LLM (GPT-4o-mini)
  5. El LLM genera una respuesta fundamentada en los chunks
  6. Se retorna la respuesta con fuentes, evidencia y latencia

CAPAS DE LA APLICACION:
  +-------------------------------------------+
  |           API REST (Controllers)          |
  |  DocumentController  |  QueryController   |
  +-------------------------------------------+
  |              Services                     |
  |  DocumentIngestionService | RAGService    |
  +-------------------------------------------+
  |              Models                       |
  |    QueryRequest    |    QueryResponse     |
  +-------------------------------------------+
  |          Spring AI Framework              |
  |   VectorStore  |  ChatClient  | Embeddings|
  +-------------------------------------------+
  |         Infrastructure                    |
  |     PostgreSQL + pgvector  |  OpenAI API  |
  +-------------------------------------------+

7.5 SERVICIO RAG: RAGService.java
--------------------------------------------------------------------------------

Paquete: com.rag.service
Anotaciones: @Service

Dependencias inyectadas:
  - VectorStore vectorStore
  - ChatClient.Builder chatClientBuilder (Spring AI - cliente del LLM)
  - @Value("${rag.top-k}") int defaultTopK

Metodo principal: query(String query, Integer topK)
  Retorna: QueryResponse

Flujo:
  1. Determina el valor de topK (usa el proporcionado o el default)
  2. Registra el timestamp de inicio para medir latencia
  3. Configura la busqueda de similitud:
     - SearchRequest con la query del usuario
     - Limita a topK resultados
  4. Ejecuta busqueda en VectorStore (pgvector)
     - Convierte la query en embedding automaticamente
     - Busca los vectores mas similares por distancia coseno
  5. Extrae los documentos recuperados con su contenido y metadatos
  6. Construye el prompt para el LLM:
     - System prompt: instrucciones para responder basandose SOLO en el contexto
     - User prompt: la pregunta del usuario + los chunks recuperados como contexto
  7. Envia al LLM (GPT-4o-mini via OpenAI API)
  8. Recibe la respuesta generada
  9. Calcula la latencia total en milisegundos
  10. Construye y retorna QueryResponse con:
      - answer: respuesta del LLM
      - sources: lista de fuentes con content y metadata
      - hasEvidence: true si se encontraron chunks relevantes
      - latencyMs: tiempo total de procesamiento

Prompt del sistema (concepto):
  "Responde la pregunta basandote UNICAMENTE en el contexto proporcionado.
   Si la informacion no esta en el contexto, indica que no tienes suficiente
   informacion. Cita las fuentes relevantes."

10. EXPERIMENTOS Y METRICAS (SPRINT 4)
================================================================================

Los experimentos del Sprint 4 evaluan el rendimiento del sistema RAG
variando parametros clave de configuracion.

EXPERIMENTO 1: Tamano de Chunk (chunk-size)
  Variable: rag.chunk-size en application.yml
  Valores probados: 300 vs 500 caracteres
  Hipotesis: Chunks mas pequenos pueden dar mayor precision pero menos contexto.
             Chunks mas grandes conservan mejor el contexto pero pueden
             introducir informacion irrelevante.
  Metrica: precision_score, recall_score

EXPERIMENTO 2: Top-K Retrieval
  Variable: rag.top-k en application.yml
  Valores probados: 3 vs 5
  Hipotesis: Mayor top-K recupera mas informacion pero puede introducir
             ruido. Menor top-K es mas preciso pero puede omitir datos.
  Metrica: precision_score, recall_score, latency_ms

EXPERIMENTO 3: Modelos de Embedding
  Variable: spring.ai.openai.embedding.options.model en application.yml
  Valores probados: text-embedding-3-small (y potencialmente otros)
  Hipotesis: Diferentes modelos de embedding generan representaciones
             vectoriales de distinta calidad semantica.
  Metrica: precision_score, recall_score

ALMACENAMIENTO DE METRICAS:
  Las metricas se almacenan automaticamente en la tabla evaluation_metrics:
    - experiment_name: identificador del experimento
    - query_text: la consulta evaluada
    - precision_score: precision del resultado (0.0 a 1.0)
    - recall_score: recall del resultado (0.0 a 1.0)
    - latency_ms: tiempo de respuesta en milisegundos
    - top_k: parametro top-K usado
    - chunk_size: tamano de chunk usado
    - created_at: timestamp del experimento

CONSULTA SQL PARA RESULTADOS:
  SELECT experiment_name,
         AVG(precision_score) AS avg_precision,
         AVG(recall_score) AS avg_recall,
         AVG(latency_ms) AS avg_latency
  FROM evaluation_metrics
  GROUP BY experiment_name;

11. ROADMAP Y PROXIMOS PASOS
================================================================================

1. ESCALAR BASE DE CONOCIMIENTO
   - Aumentar de documentos actuales a 30-50 PDFs
   - Diversificar fuentes para robustecer las respuestas
   - Probar con documentos de mayor longitud y complejidad

2. INTERFAZ DE USUARIO (UI)
   - Implementar frontend interactivo con Thymeleaf o React
   - Permitir consultas en tiempo real desde navegador web
   - Mostrar fuentes y evidencia de forma visual

3. RE-RANKING
   - Implementar algoritmo de re-clasificacion post-retrieval
   - Reordenar chunks por relevancia real antes de enviar al LLM
   - Mejorar la calidad de las respuestas finales

4. BUSQUEDA HIBRIDA
   - Combinar busqueda vectorial (semantica) con BM25 (palabras clave)
   - Capturar tanto similitud semantica como coincidencia lexica
   - Mejorar recall para consultas con terminos tecnicos especificos

5. MEMORIA DE CONVERSACION
   - Mantener contexto entre preguntas del mismo usuario
   - Permitir dialogos naturales multi-turno
   - Implementar historial de conversacion con ventana de contexto
